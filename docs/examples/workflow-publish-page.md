# Example Workflow: Publish Page

This example shows a practical Dify Workflow built around the implemented `publish_page` tool.
The goal is to draft content, route it through human review when needed, and then create or update a BookStack page without hardcoding any real BookStack or Dify environment values.

## Recommended Node Order

1. **Start / Trigger**
   - Receive the business request, source content, or structured page metadata.
2. **Draft Generation**
   - Use an LLM node or upstream system node to produce the page title and Markdown body.
3. **Structured Field Mapping**
   - Normalize workflow variables into the exact `publish_page` input fields.
4. **Human Review / Approval**
   - Let an operator confirm the title, Markdown, target location, and publishing intent.
5. **Optional Branch: Known Page vs Unknown Page**
   - If you already know the target `page_id`, pass it directly.
   - If you do not know the `page_id`, let `publish_page` decide whether to update an exact title match or create a new page.
6. **BookStack `publish_page` Tool Node**
   - Submit the final page payload.
7. **Result Router**
   - Branch on success or failure.
8. **Post-Publish Actions**
   - Store `page_id`, send notifications, write audit records, or continue downstream automation.

## Key Input Fields

The current `publish_page` tool expects these practical inputs:

- `title`: required, trimmed string.
- `markdown`: required, trimmed string.
- `page_id`: optional. When present, the tool updates that page directly.
- `book_id`: optional for updates, but required for creation if `chapter_id` is not provided.
- `chapter_id`: optional for updates, but required for creation if `book_id` is not provided.
- `tags`: optional. Pass only when your workflow actually has tags to write.

### Minimal Update Payload

Use this when your workflow already knows the page to update:

```json
{
  "title": "Scheduler Design",
  "markdown": "# Scheduler Design\n\nApproved content...",
  "page_id": "123"
}
```

### Create-Or-Update Payload by Title

Use this when your workflow knows the target Book or Chapter, but not a page ID:

```json
{
  "title": "Scheduler Design",
  "markdown": "# Scheduler Design\n\nApproved content...",
  "book_id": "1",
  "chapter_id": "12",
  "tags": "status:approved"
}
```

## Create / Update Logic

The implemented `publish_page` behavior is:

1. If `page_id` is provided, the tool updates that page directly.
2. If `page_id` is not provided, the tool searches existing pages by `title`.
3. It keeps only exact title matches, then narrows them by `book_id` and `chapter_id` when those fields are present.
4. If exactly one match remains, the tool updates that page.
5. If more than one exact match remains, the tool returns `success=false` with `error=ambiguous title match`.
6. If no exact match remains, the tool creates a new page.
7. For new page creation, at least one of `book_id` or `chapter_id` must be provided, otherwise the tool returns `success=false` with `error=book_id or chapter_id is required to create a new page`.

This means a workflow designer should prefer `page_id` whenever a stable target page is already known. When `page_id` is not available, include `book_id` and/or `chapter_id` to reduce accidental title collisions and make creation possible.

## Human Review / Approval Suggestions

Before the `publish_page` node, add a human approval step when any of these apply:

- The page content is generated by an LLM and must be checked for accuracy.
- The workflow may overwrite an existing BookStack page.
- The page belongs to a controlled book, chapter, or team-owned documentation area.
- The Markdown contains release notes, policy text, or operational procedures.

Recommended review checklist:

- Confirm `title` is final and not likely to collide with another page.
- Confirm `markdown` is complete and ready for publication.
- Confirm `page_id` is correct when doing a known update.
- Confirm `book_id` and `chapter_id` point to the intended destination.
- Confirm the workflow should publish now rather than queue for later.

## Success Output and Downstream Usage

On success, the tool returns JSON with this shape:

```json
{
  "success": true,
  "page_id": 123,
  "title": "Scheduler Design",
  "url": "https://bookstack.example/books/example/page/scheduler-design",
  "action": "created",
  "raw": {}
}
```

`action` is either:

- `created`
- `updated`

Useful downstream patterns:

- Use `page_id` as the stable identifier for future update workflows.
- Use `url` in notification, approval-complete, or audit log messages.
- Use `action` to branch between “new page published” and “existing page updated” notifications.
- Use `title` when composing human-readable summaries.
- Store `raw` only if a later node needs BookStack response details; otherwise prefer the normalized fields above.

## Error Handling Suggestions

On failure, the tool returns a text payload in this pattern:

```text
success=false
error=<message>
```

Recommended workflow handling:

- Add a router after the tool node that checks whether the publish step succeeded.
- Send user-correctable input errors to a human repair branch.
- Send service or permission failures to an operator alert branch.
- Persist the attempted title, destination fields, and error message for troubleshooting.

Common errors to handle explicitly:

- `title is required`
- `markdown is required`
- `book_id or chapter_id is required to create a new page`
- `ambiguous title match`
- `Invalid credentials`
- `Permission denied`
- `Book not found`
- `Chapter not found`
- `Page not found`
- `BookStack API unavailable`
- `Invalid BookStack response`

Suggested branching strategy:

- Validation errors: route back to an editing or approval node.
- `ambiguous title match`: ask a human to provide `page_id`, `book_id`, or `chapter_id`.
- Permission or credential errors: stop and notify the workflow operator.
- Availability or response errors: retry conservatively or queue for manual retry, depending on your environment.

## Example End-to-End Workflow Idea

An effective Dify Workflow can follow this sequence:

1. Accept a documentation request.
2. Generate a Markdown draft.
3. Extract `title`, `markdown`, and destination metadata into structured variables.
4. Pause for human approval.
5. Call `publish_page`.
6. If `action=created`, record the new `page_id` for future updates.
7. If `action=updated`, notify subscribers that an existing page changed.
8. If publish fails, branch to manual correction or operator escalation.

This pattern keeps the content-generation step, approval step, and BookStack write step clearly separated, which makes the workflow easier to audit and safer to operate.
